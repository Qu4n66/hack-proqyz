1. WHAT THIS PROJECT IS
================================================================================

A Node.js + Playwright CLI that automates uploading IELTS Reading quizzes
into ProQyz (app.proqyz.com). You provide structured JSON; the tool drives
the ProQyz UI end-to-end.

NOT a content generator — a content UPLOADER.

Tech stack:
  - Node.js >= 20, ES Modules
  - Playwright (browser automation)
  - Zod (schema validation)
  - pino (structured logging)
  - @inquirer/prompts (interactive CLI)

Total codebase: ~11,400 lines of JavaScript across 21 source files.

7. KNOWN PROBLEMS AND WHY CERTAIN THINGS CAN'T BE IMPLEMENTED
================================================================================

Problem 1: UiQuizUploader.js is 6,290 lines
─────────────────────────────────────────────
This single file handles ALL ProQyz UI interaction: quiz creation, passage
creation, 4 question types, tab navigation, modal management, passage
selection, type picking, content writing, answer verification, checkpoint
resume, failure capture, and diagnostic dumping. It is the biggest risk
in the project.

Impact: Any change risks regressions in unrelated flows. Debugging is
slow because the failure surface is enormous. Adding a new question type
requires threading through ~20 private methods.

Why it's hard to fix: The file grew organically from each real ProQyz run
discovering new UI behaviors. Every modal, tab, and form had different
selector shapes and timing characteristics. The "fix it properly" path
would require splitting into separate modules (QuizCreationDriver,
PassageDriver, FillUpDriver, RadioDriver, etc.) — a significant refactor.


Problem 2: Selector brittleness
───────────────────────────────
ProQyz's UI has no data-testid attributes. Every selector is a
best-guess combination of CSS classes, ARIA roles, and visible text.
When ProQyz updates its UI, selectors break silently.

Impact: 46 of the Cam17 upload failures are from selectors not matching
the current ProQyz DOM.

Why it's hard to fix: Without stable test IDs, the only reliable
approach is (a) a full recon pass after every ProQyz deploy, or
(b) a more resilient multi-signal approach (which is already attempted
but adds complexity and timing sensitivity).


Problem 3: TinyMCE timing issues
─────────────────────────────────
The passage content and question content are written into TinyMCE editors.
TinyMCE's iframe body is not immediately available after the editor loads,
and the `setContent()` API fires events asynchronously. Multiple attempts
have been made (API path, keyboard fallback, iframe body probe), but
the timing remains fragile.

Impact: Many Cam17 failures are from content not being "seen" in the
editor even though it was written correctly — the iframe body probe
reads from the wrong frame.

Why it's hard to fix: TinyMCE's internal state management is opaque.
The editor fires `change` and `save` events, but React's controlled
component state may not have updated by the time we read back. There
is no reliable way to force a synchronous state update.


Problem 4: Checkbox upload not implemented
──────────────────────────────────────────
The addQuestion method has an explicit `throw` for checkbox. The domain
schema fully supports checkbox (options + answers array). The
radioStrategy.js already has `selectCorrectCheckboxes`. But the UI
driver has no `_fillCheckboxQuestion` method.

Why: The project prioritized fill_up → select → radio. Checkbox was
"intentionally deferred" because the ProQyz checkbox editor has not
been fully probed yet — the selector for checkbox correct-answer
markers is best-guess only.

What would be needed:
  1. Create _fillCheckboxQuestion() in UiQuizUploader.js
  2. It would be similar to _addOptionsQuestion(q, "checkbox")
  3. Use selectCorrectCheckboxes from radioStrategy.js
  4. Test against real ProQyz UI to verify selectors


Problem 5: Grouped fill_up answer list verification
───────────────────────────────────────────────────
The _fillGroupedAnswerList method has 3 strategies:
  1. Badge auto-extract (primary)
  2. Per-blank inputs (fallback)
  3. Newline-joined textarea (last resort)

All 3 have been implemented but 2 failure captures show the badge
count never reaching the expected number. The TinyMCE brace-scan may
need a longer settle time, or the selector `.badge.bg-success` may
not match the current ProQyz markup.


Problem 6: No listening support (Phase 3)
──────────────────────────────────────────
The schema supports listening quizzes (ListeningQuizSchema), but:
  - listeningInput.js is a stub
  - The pipeline run.js explicitly throws for non-reading quizzes
  - No audio file upload logic exists
  - No section-based question flow

Why deferred: Listening requires audio file upload via file input,
which is a fundamentally different flow from the text-based
Reading upload.


Problem 7: Raw text parser not implemented (Phase 2)
─────────────────────────────────────────────────────
rawTextParser.js is a stub that throws "not implemented yet (Phase 2)".
Currently, users must prepare JSON manually or via external AI/parser.

Why deferred: The JSON input path works and the parser is a separate
workstream. It was lower priority than getting the uploader reliable.

8. WHAT THE FAILURES TELL US
================================================================================

The 112 failure artifacts reveal a pattern:

  Phase 1 (Jun 9, early):  Type-pick failures — the react-select
  dropdown for question type wasn't committing the selection.
  The modal stayed on the Basic tab. Fixed by adding Enter press
  + click-outside commit.

  Phase 1 (Jun 9, mid):    Answer-list probe failures — grouped
  fill_up badges not appearing. The TinyMCE brace-scan needs a
  blur event to fire. Fixed by adding Tab press after insertText.

  Phase 1 (Jun 9, late):   Radio MCQ failures — option rows not
  rendering, DOM mismatches. The radio editor has a different DOM
  shape than fill_up/select. Partially fixed but still fragile.

  Phase 2 (Jun 10):        Continued Radio MCQ + Cam17 failures.
  The patterns repeat: options too few, dom-mismatch, row-not-created.
  This suggests the radio editor's "Add Option" button behavior
  differs from what the code expects.

  Ongoing:                 Passage content verification false
  negatives — the TinyMCE iframe probe reads from the wrong frame
  or the body hasn't rendered yet. The code has multiple fallback
  strategies but they all rely on timing.

11. SUMMARY
================================================================================

The project is a well-architected Playwright automation tool that has
successfully implemented the core pipeline: JSON → validate → normalize →
create quiz → add passages → add questions → save.

WHAT WORKS:
  - Schema validation (solid, 20+ unit tests)
  - Fill-up upload (working)
  - Select upload (working)
  - Login/session management (working)
  - Checkpoint/resume (working)
  - Dry-run mode (working)

WHAT'S FRAGILE:
  - Radio MCQ upload (works sometimes, 46 failures)
  - Cam17 full upload (timing-dependent, 46 failures)
  - TinyMCE content verification (iframe timing)
  - Passage selection (multi-strategy probe)

WHAT'S MISSING:
  - Checkbox upload (throws "not supported")
  - Raw text parser (Phase 2 stub)
  - Listening support (Phase 3 stub)
  - E2E tests for all question types
  - Modular file structure (everything in one 6,290-line file)

WHY CERTAIN THINGS HAVEN'T BEEN DONE:
  1. The ProQyz UI is a moving target — selectors break between deploys
  2. TinyMCE's iframe timing is inherently unreliable for automation
  3. The project prioritized making fill_up/select/radio work reliably
     over adding checkbox support
  4. The codebase grew from real-world debugging sessions, leading to
     a monolithic uploader file that resists safe modification
